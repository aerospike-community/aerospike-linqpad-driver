# Aerospike LINQPad AI Features

> Pre-release preview for AI-assisted Aerospike query generation in the Aerospike LINQPad Driver.

The Aerospike LINQPad AI features make it easier to move from a natural-language request to runnable LINQPad C# Statements code. The AI context generated by the driver gives LINQPad AI enough information to understand the active Aerospike connection, generated namespaces and sets, observed bins, Auto-Value behavior, and the difference between LINQPad-driver code and the native Aerospike C# client API.

The goal is simple:

> Describe the Aerospike data question you want answered, then generate useful C# for LINQPad.

---

## What This Enables

The AI features can help generate or explain:

- LINQ query-syntax scripts against generated LINQPad driver sets.
- LINQ method-syntax scripts when preferred.
- AValue-aware filters and projections for missing, null, sparse, or mixed-type bins.
- Complex joins across sets such as `Customer`, `Invoice`, `InvoiceLine`, `Track`, `Album`, and `Artist`.
- Native Aerospike C# client API code.
- Server-side Aerospike filter expressions.
- Nested CDT/list/map expression filters.
- Explanations of existing LINQPad Aerospike queries.
- New `.linq` query files based on the generated AI response.

---

## Why This Matters

Aerospike is schemaless, and LINQPad is an interactive exploration environment. That combination is powerful, but code generation needs more context than a normal C# prompt.

The AI context helps the model understand:

- Which namespaces and sets exist in the active connection.
- Which bins and generated properties are available.
- Whether Auto Values are enabled.
- How to use `AValue`, `APrimaryKey`, and generated properties correctly.
- When to use client-side LINQ versus server-side Aerospike expressions.
- When to use the native Aerospike C# client API.
- How to avoid unsafe casts, missing-bin errors, and mixed-type assumptions.

## Natural-Language Request Examples

### Generate a LINQ query with joins

```text
I would like a list of customers with invoices and associated artists, albums,
and track names they purchased, including subtotals.

Please use LINQ query syntax.
```

Expected output:

- LINQPad-driver mode.
- Generated sets and properties.
- Query-syntax joins.
- `.AsEnumerable()` where collection-style LINQ is required.
- A bounded result set using `Take(...)`.
- Output with `Dump()`.

---

### Explain an existing LINQPad Aerospike query

```text
Explain this LINQPad Aerospike query.

Query:

from customer in test.Customer.AsEnumerable()
where customer.FirstName.TryApply<string, bool>(name => name.StartsWith("J"))
select customer

Focus on:
- Whether this is client-side LINQ or server-side Aerospike expression logic.
- Why AsEnumerable() is used.
- How AValue and TryApply affect null, missing, and mixed-type values.
- Whether generated properties are used correctly.
- Any safety or performance considerations.
```

Expected output:

- Explanation that this is client-side LINQ after records are materialized.
- Explanation that `AsEnumerable()` enables normal LINQ-to-Objects behavior.
- Explanation that `TryApply<string, bool>(...)` safely handles null, missing, and mixed-type values.
- Notes about when server-side expressions may be more efficient.

---

### Translate LINQPad-driver code to native Aerospike API

```text
Query:

from customer in test.Customer.AsEnumerable()
where customer.FirstName.TryApply<string, bool>(name => name.StartsWith("J"))
select customer

I want to translate this to use the Aerospike native API with server-side expressions.
```

Expected output:

- Native Aerospike C# client API code.
- `AerospikeClient`, `ClientPolicy`, `ScanPolicy` or `QueryPolicy`.
- Raw namespace, set, and bin names.
- `record.GetValue("FirstName")` for returned records.
- `Exp.Build(...)` assigned to the native policy.
- No generated LINQPad-driver sets, `SetRecords`, `AValue`, `PK`, or generated record properties.

---

### Generate native API code for nested CDT server-side expressions

```text
I want to obtain all customer records from the CustInvDoc set for TrackIds
2955, 1447, 179, or 3169.

"TrackId" is within the "Lines" map, which is within the "Invoices" map.

I only need the customer record without invoices, plus the matching TrackIds
with the associated artist name and album title.

Please generate this using the Aerospike native API with server-side expressions.
```

Expected output:

- Native Aerospike C# client API code.
- A server-side expression using nested CDT traversal.
- `CDTExp.SelectByPath(...)` and `CTX` selectors for `Invoices[*].Lines[*].TrackId`.
- `ListExp.GetByValue(ListReturnType.EXISTS, ...)` for target TrackId matching.
- Enrichment from related sets such as `Track`, `Album`, and `Artist`.
- Returned customer fields without the full `Invoices` payload.

## LINQPad-Driver Mode Versus Native API Mode

### LINQPad-driver mode

Use this for most interactive exploration:

```csharp
var rows =
    (from customer in test.Customer.AsEnumerable()
     where customer.FirstName.TryApply<string, bool>(name => name.StartsWith("J"))
     select customer)
    .Take(100);

rows.Dump();
```

LINQPad-driver mode can use:

- Generated namespaces and sets.
- Generated record properties.
- `AValue` and `APrimaryKey`.
- Driver extension methods.
- `SetRecords.Query(...)` for driver-side expression filters.

### Native Aerospike C# client API mode

Use this when the user asks to generate native API code or server-side native expressions:

```csharp
using Aerospike.Client;

var policy = new ScanPolicy
{
    filterExp = Exp.Build(
        Exp.RegexCompare("^J", RegexFlag.NONE, Exp.StringBin("FirstName")))
};
```

Native API mode uses:

- `AerospikeClient`.
- `ClientPolicy`, `ScanPolicy`, `QueryPolicy`, and `Statement`.
- Raw namespace, set, and bin names.
- `record.GetValue("BinName")`.
- `Exp.Build(...)` for native policy expressions.

Native API mode should not use generated LINQPad-driver sets or `AValue`.

## Auto Values and AI-Generated Code

Auto Values are central to AI-generated LINQPad-driver code.

When Auto Values are enabled, generated record properties may be represented as `AValue` instances. AI-generated code should avoid unsafe casts and instead use AValue-aware helpers.

Common examples:

```csharp
customer.FirstName.TryApply<string, bool>(name => name.StartsWith("J"))
```

```csharp
customer.Total.CanConvert<decimal>() && customer.Total.Convert<decimal>() > 10
```

```csharp
!customer.Company.IsEmpty
```

```csharp
line.TryGetValue("TrackId", AValue.Empty)
```

These patterns make generated code safer when records are sparse, bins are missing, or values are mixed-type across records.

For more detail, see [`README_AVALUES.md`](./README_AVALUES.md).

## Safety Notes

Generated code should still be reviewed before execution.

Important safety expectations:

- Default to read-only, bounded queries.
- Use `Take(...)` or filters for exploratory queries.
- Ask before destructive operations.
- Prefer generated properties over string-indexer access when available.
- Use AValue helpers for null, missing, or mixed-type values.
- Use server-side expressions when the user explicitly asks for server-side filtering.
- Use native API only when the user asks for it or when the LINQPad-driver API is not appropriate.
